# 보안 기초

## 보안의 2가지 종류

- 기밀성: 메세지 내용을 보내는 사람과 받는 사람 이외에는 알 수 없게 하는 것
- 무결성: 받는 사람 입장에서 전달받은 메세지의 내용이 위조되지 않았다고 확신할 수 있는 것

## 보안 도구 세트

- 암호화: 복호화가 가능한 변환
  - 보안 도구1: 대칭키 생성 알고리즘
    - 대칭키를 생성하는 알고리즘
    - 대칭키 알고리즘: DES, 3DES, AES
    - 대칭키 알고리즘 옵션: ECB, CBC
    - 예시: 3DES-CBC, AES-CBC
    - 속도가 매우 빠르다.
  - 보안 도구2: 비대칭키 생성 알고리즘
    - 개인키와 공개키 쌍을 생성하는 알고리즘
    - 비대칭키 알고리즘: RSA
    - 속도가 매우 느리다
- 보안 도구3: 해싱
  - 복호화가 불가능한 변환
  - 비밀번호를 저장할 때는 역변환이 불가능해야 하므로 암호화가 아닌 해싱 과정을 거쳐야 합니다.
  - 해싱 알고리즘: SHA256
  - 무차별 대입 공격을 막기 위해서 랜덤 솔트가 추가되어야 더 안전하다.
  - 무차별 대입 공격을 막기 위해서 해싱 적용 횟수를 적당히 늘려야 더 안전하다.
  - 대표적인 라이브러리: bcrypt

## 해커의 공격 루트 1번: 클라이언트와 서버 사이의 통신

- https를 사용하면 암호화된 통신을 할 수 있습니다.
- https
  - 서버1: 개인키, 공개키 쌍을 만들고 공개키에 기관의 인증을 받아서 인증서 생성
  - 클라이언트1: 대칭키(일회용)를 만들고 서버의 공개키를 받아서 암호화한 후에 서버에 전송합니다.
  - 서버2: 암호화된 대칭키(일회용)를 서버의 개인키로 풀어서 임시 보관합니다.
  - 이 순간, 서버와 클라이언트가 모두 대칭키(일회용)를 공유하게 되었습니다.
  - 위에서 설명한 4가지 도구(공개키, 개인키, 대칭키, 해싱)를 적당히 사용해서 지금부터 암호화된 통신이 가능합니다.
  - 해당 통신이 종료되면 대칭키를 폭파시킵니다.

## 해커의 공격 루트 2번: 서버 접근 권한을 획득

- 회사 내의 직원의 이메일 등을 통해서 개발 노트북에서 악성 코드를 실행하는 방식으로 접근 권한을 확보

## 해커의 공격 루크 3번: 자격 증명을 획득

- 자격 증명이란?
  - 토큰
    - 형식: 헤더.데이터.서명
    - 저장 및 관리
      - 브라우저: 로컬 스토리지에 저장하고 관리합니다.
      - 앱: 파일 시스템에 저장하고 관리합니다.
    - 전송
      - 헤더에 아래의 키 값 쌍을 추가해서 전송합니다.
      - Authorization: Bearer [토큰]

  - 쿠키
    - 형식: 이름=데이터, 이름=데이터.서명
    - 저장 및 관리
      - 브라우저: 쿠키 스토리지에 의해 자동으로 저장되고 관리됩니다.
    - 전송
      - 브라우저에 의해 대부분 자동으로 전송됩니다.

- CSRF 공격
  - 공격 발생 위치: evil.com
  - 공격자의 목표: 구글 사용자 도용
  - 공격자: evil.com 프론트엔드에서 구글 서버로 비밀번호 변경을 요청하는 악성 스크립트를 구현합니다.
  - 사용자: 구글에 로그인이 되어있는 상황에서, evil.com에 접속하면 악성 스크립트가 실행되어서, 비밀번호 변경을 요청하는 구글 서버로 요청이 전송될 때, 브라우저가 자동으로 구글 전용 쿠키 스토리지를 찾아서 쿠키를 첨부해서 서버로 전송한다.

- CSRF 방어
  - 방어1: evil.com에 접속하지 않습니다.
  - 방어2: evil.com에 접속해도 자격 증명에 접근할 수 없도록 합니다.
    - SOP: 구글에서 발급해준 자격 증명 토큰은 구글 전용 로컬 스토리지에 저장되기 때문에, evil.com에 접속해도 js가 자격 증명 토큰에 접근할 수 없습니다.
    - 토큰을 서버에 전송하는 것은 js로 헤더에 직접 추가해야 하기 때문에, evil.com은 정상적으로 요청을 실행할 수 없습니다.

    - SOP: 구글에서 발급해준 자격 증명 쿠키는 구글 전용 쿠키 스토리지에 저장되기 때문에, evil.com에 접속해도 js가 자격 증명 쿠키에 접근할 수 없습니다.
    - 그러나, 쿠키를 서버에 전송하는 것은 브라우저가 자동으로 첨부하는 것이 기본 세팅이기 때문에, evil.com은 정상적으로 요청을 도용해서 실행할 수 있습니다.
  - 방어3
    - SOP만으로는 쿠키 자동 전송 문제를 막을 수 없기 때문에, 쿠키를 발급하는 쪽(구글)에서 쿠키에 정교하게 세팅을 해서 브라우저가 자동으로 전송하는것을 막아야 합니다.
    - sameSite 옵션을 사용합니다.
      - strict: 도메인 출처가 다르면 절대로 쿠키를 첨부하지 않음.
      - lax: 도메인 출처가 다르면, 기본 요청은 쿠키 동봉을 허락하고 위험한 요청은 쿠기 첨부를 불허함
      - none: 도메인 출처가 같든 다르든, 쿠키를 언제나 첨부함

- XSSI 공격
  - 공격 발생 위치: evil.com
  - 공격자의 목표: 구글 사용자 데이터 읽기
  - 공격자: evil.com 프론트엔드에서 구글 서버로 유저 정보를 요청하는 악성 스크립트를 구현합니다.
  - 사용자: 구글에 로그인이 되어있는 상황에서, evil.com에 접속하면 악성 스크립트가 실행되어서, 유저 정보를 요청하는 구글 서버로 요청이 전송될 때, 브라우저가 자동으로 구글 전용 쿠키 스토리지를 찾아서 쿠키를 첨부해서 서버로 전송한다.

- XSSI 방어
  - 방어1: evil.com에 접속하지 않습니다.
  - 방어2: SOP 정책으로 막아줍니다.
  - 방어3
    - SOP만으로는 쿠키 자동 전송 문제를 막을 수 없기 때문에, 쿠키를 발급하는 쪽(구글)에서 쿠키에 정교하게 세팅을 해서 브라우저가 자동으로 전송하는것을 막아야 합니다.
    - sameSite 옵션을 사용합니다.
      - strict: 도메인 출처가 다르면 절대로 쿠키를 첨부하지 않음.
      - lax: 도메인 출처가 다르면, 기본 요청은 쿠키 동봉을 허락하고 위험한 요청은 쿠기 첨부를 불허함
      - none: 도메인 출처가 같든 다르든, 쿠키를 언제나 첨부함
    - csrf state 검증
    - 더블 쿠키 패턴
  - 방어4
    - CORS로 방어가 됩니다.

- CORS란 무엇일까?
  - 기본적으로 브라우저는 프론트엔드와 백엔드가 서로 출처가 다를 경우 서버에서 요청을 보내는 것을 허락(브라우저 요청)하지만 그 응답을 읽는것을 차단합니다.
  - 서버가, 우리 데이터를 볼 수 있는 허락된 출처를 선별해서 화이트리스트로 등록합니다.
  - 브라우저가 화이트리스트를 체크해서 허락된 프론트엔드라면 데이터를 읽을 수 있도록 통과시켜줍니다.
    - Access-Control-Allow-Origin: 프론트엔드출처
    - 위 작업이 너무나도 귀찮기 때문에, cors라이브러리를 사용한다.
  - 기본적으로 브라우저는 자격 증명을 서로 다른 출처끼리 요청하고 받는것을 금지합니다. 그렇기 때문에, 서버가 위에서 정의한 화이트리스트 목록들은 자격 증명을 받을 수 있도록 화이트리스트에 추가해주면 브라우저가 자격 증명을 통과시켜서 저장해줍니다.
  - 클라이언트가 브라우저 요청이 아니라면, 스크립트 요청에서는 반드시 withCredentials: true 옵션을 추가해야 합니다.

- XSS 공격
  - 공격 발생 위치: 우리 웹사이트
  - 공격자의 목표: 자격 증명 탈취
  - 공격자: 우리 웹사이트의 특정 페이지에 악성 스크립트를 삽입합니다.
  - 사용자: 우리 웹사이트에 접속해서 인증을 완료하고 자격 증명을 브라우저에 저장합니다.
  - 사용자: 우리 웹사이트의 특정 페이지에 접근하면 악성 스크립트가 실행되어서 자격 증명이 탈취됩니다.

- XSS 방어
  - 방어1: 악성 스크립트가 우리 웹사이트에서 실행될 수 없게 만들기: 출력값 이스케이핑(React를 사용하면 해결)
  - 방어2: 이 페이지는 서버가 허락한 스크립트만 실행을 허용한다고 적어서보내기: CSP(콘텐츠 시큐리티 폴리시)
  - 방어3: httpOnly 옵션이 true인 쿠키를 클라이언트에게 전송해서 js로 쿠키를 탈취할 수 없도록 만들자. 토큰은 포기하자.

- 우리의 결론
  - 일단은 토큰을 사용하자.
